# Production Queue Structure Guide

**Created:** 2026-02-13
**Last Updated:** 2026-02-13
**Purpose:** Multi-step workflow pattern for standardized processes
**Status:** Final

---

## 1. Purpose

The production queue folder structure enables seamless parallel work between human operators and AI agents by providing:

- **Clear visibility** into work status at a glance
- **Non-blocking workflows** where humans and AI work on different files simultaneously
- **Organized progression** from generation through publication
- **Easy orchestration** for automated status tracking and triggering
- **Batch management** for handling multiple files efficiently
- **Regeneration paths** when content needs revision

This system transforms content creation from sequential (AI generates → human reviews → publish) to parallel (AI generates while humans review previous batch, etc.).

## 2. Complete Folder Structure

```
6_Session_Working_Files/
 {project-name}/
     production-queue.md          # Central status tracking file
     0-specs/                     # Reference materials for all agents
        project-brief.md
        content-requirements.md
        style-guide.md
        examples/
    
     1-generating/                # Currently being generated by AI
        article-001.md
        article-002.md
        batch-metadata.json
    
     2-ready-for-review/          # Awaiting human QC
        article-003.md
        article-004.md
        review-queue.json
    
     3-approved/                  # Passed QC, awaiting media
        article-005.md
        article-006.md
        approval-log.json
    
     4-with-media/                # Media added, ready to publish
        article-007.md
        article-007-media/
           images/
           videos/
           manifest.json
        media-log.json
    
     5-published/                 # Live/published content
         article-008.md
         article-009.md
         archived/                 # Previous versions if updated
         published-log.json
```

### Folder Descriptions

| Folder | Purpose | Owner | Duration |
|--------|---------|-------|----------|
| **0-specs/** | Single source of truth for requirements, style, examples | Human/Admin | Persistent |
| **1-generating/** | AI actively working on content generation | AI Agent | Until generation complete |
| **2-ready-for-review/** | Awaiting human review, QC, approval | Human Reviewer | 1-3 hours typically |
| **3-approved/** | Passed review, queued for media team | Media Coordinator | Until media added |
| **4-with-media/** | Complete content with media, final checks | QA/Publisher | Before publishing |
| **5-published/** | Live on platform; archive of published work | System | Permanent |

## 3. What Goes in Each Folder (With Examples)

### 0-specs/ (Reference)

**Purpose:** Immutable source of truth for all content generation

**Contents:**
- `project-brief.md` - Overall project goals and scope
- `content-requirements.md` - What each content piece must include
- `style-guide.md` - Tone, formatting, technical standards
- `examples/` - Reference pieces showing desired quality
- `audience-profile.md` - Target reader characteristics
- `seo-guidelines.md` - SEO keywords, meta rules (if applicable)
- `compliance-checklist.md` - Legal/brand requirements

**Example content-requirements.md:**
```markdown
# Content Requirements - Blog Articles

## Article Structure
- Title (60-70 chars)
- Meta description (150-160 chars)
- Introduction (100-150 words)
- 3-5 main sections
- Conclusion
- 1-2 related links

## Technical Requirements
- Level: Intermediate (explain jargon once)
- Length: 1,200-1,500 words
- Includes: 1 code example, 1 comparative table
- No: First-person singular, excessive links

## Quality Standards
- Grammar: Grade 9 reading level
- Tone: Authoritative but approachable
- Accuracy: Fact-check all claims with sources
```

### 1-generating/ (Active Work)

**Purpose:** AI content currently being created

**Contents:**
- Individual content files (markdown, doc, etc.)
- `batch-metadata.json` - Tracking batch progress
- Partially completed files (safe to overwrite)

**Example batch-metadata.json:**
```json
{
  "batch_id": "batch-2025-02-10-001",
  "total_items": 5,
  "generation_started": "2025-02-10T09:00:00Z",
  "items": [
    {
      "filename": "article-001.md",
      "spec_id": "req-blog-ai-safety",
      "status": "generating",
      "started": "2025-02-10T09:05:00Z",
      "estimated_completion": "2025-02-10T09:25:00Z",
      "ai_agent": "ContentGenerator-v2"
    },
    {
      "filename": "article-002.md",
      "spec_id": "req-blog-ml-trends",
      "status": "queued",
      "priority": 1
    }
  ]
}
```

**Best Practice:** Files here can be regenerated/overwritten by AI. Don't make manual edits here without coordinating with AI agent.

### 2-ready-for-review/ (Pending QC)

**Purpose:** Complete content awaiting human review and approval

**Contents:**
- Final generated files (ready to review)
- `review-queue.json` - Review task assignments
- Reviewer notes (as comments or separate files)

**Example review-queue.json:**
```json
{
  "batch_id": "batch-2025-02-10-001",
  "review_started": "2025-02-10T10:00:00Z",
  "items": [
    {
      "filename": "article-001.md",
      "submitted_for_review": "2025-02-10T09:30:00Z",
      "assigned_to": "reviewer-sarah",
      "assigned_date": "2025-02-10T10:00:00Z",
      "deadline": "2025-02-10T12:00:00Z",
      "status": "in_review"
    },
    {
      "filename": "article-002.md",
      "submitted_for_review": "2025-02-10T09:35:00Z",
      "status": "waiting_assignment"
    }
  ]
}
```

**Best Practice:** Don't delete files here. Move to 3-approved/ or back to 1-generating/ with notes.

### 3-approved/ (QC Passed)

**Purpose:** Content that passed review, waiting for media resources

**Contents:**
- Approved markdown/documents (finalized text)
- `approval-log.json` - Who approved what and when
- Comments/feedback resolutions documented

**Example approval-log.json:**
```json
{
  "batch_id": "batch-2025-02-10-001",
  "items": [
    {
      "filename": "article-001.md",
      "reviewed_by": "reviewer-sarah",
      "reviewed_date": "2025-02-10T10:45:00Z",
      "status": "approved",
      "revisions_requested": 0,
      "approved_date": "2025-02-10T10:45:00Z",
      "move_to_media_date": "2025-02-10T11:00:00Z"
    }
  ]
}
```

**Best Practice:** Files here are finalized. Changes only with approval from QC person.

### 4-with-media/ (Media Complete)

**Purpose:** Content with all media assets, ready to publish

**Contents:**
- Final content files
- `{filename}-media/` directories with images, videos, etc.
- `media-log.json` - What media was added, where, by whom
- `{filename}-media/manifest.json` - Asset manifest

**Directory structure example:**
```
4-with-media/
 article-001.md
 article-001-media/
    manifest.json
    images/
       hero-image.jpg (1200x600px, optimized)
       chart-01.png
       screenshot-01.png
    videos/
       demo-video.mp4 (H.264, max 100MB)
    alt-text.json

```

**Example manifest.json:**
```json
{
  "article": "article-001.md",
  "media_added_by": "media-coordinator-john",
  "media_added_date": "2025-02-10T13:00:00Z",
  "assets": [
    {
      "file": "images/hero-image.jpg",
      "type": "image",
      "size_mb": 0.8,
      "dimensions": "1200x600",
      "alt_text": "AI safety concept illustration",
      "usage": "Hero image (top of article)"
    },
    {
      "file": "images/chart-01.png",
      "type": "image",
      "size_mb": 0.2,
      "alt_text": "Chart showing AI safety research timeline",

  "qa_checklist": {
    "images_optimized": true,
    "alt_text_complete": true,
    "links_verified": true,
    "final_review": "passed"
  }
}
```

**Best Practice:** Don't add media after files move here without strong reason. Coordinate with media team.

**Contents:**

### Why This Works

- **At-a-glance status:** All stakeholders see where things stand
- **No meeting needed:** Updates pull automatically or on manual trigger
- **Bottleneck visibility:** Blockers appear immediately
- **Audit trail:** Who did what and when is recorded
- **Orchestrator-friendly:** Can be read/parsed by automation

## 5. File Naming Conventions

Consistent naming enables easy tracking and automation.

### Standard Format

```
{content-type}-{sequence-number}.{extension}

Examples:
- article-001.md
- article-002.md
- guide-001.md
- tutorial-001.md
- case-study-001.md
```

### Optional Extended Format (for larger projects)

```
{content-type}-{category}-{sequence}.{extension}

Examples:
- article-ai-001.md (AI category)
- article-ml-002.md (ML category)
- guide-python-001.md
- guide-javascript-002.md
```

### Metadata in Filename (when needed)

```
{content-type}-{category}-{sequence}-{date}.{extension}

Example:
- article-ai-001-20250210.md

Or in a .meta file:
article-001.meta.json
{
  "id": "article-001",
  "type": "blog-article",
  "category": "ai-safety",
  "seq": 1,
  "spec_ref": "req-blog-ai-safety",
  "created": "2025-02-10T09:05:00Z"
}
```

### Best Practices

- **Use zero-padded numbers** (001, 002, not 1, 2) for sorting
- **Keep names lowercase** for consistency and cross-platform compatibility
- **Use hyphens, not underscores or spaces** for readability
- **No version numbers in filename** (use archive folders instead)
- **Match spec_id to requirements** for easy traceability

## 6. Status Tracking Methodology

### Three Levels of Tracking

#### Level 1: Folder Location (Simplest)
File location indicates status automatically.
- In 1-generating/ → Status is "generating"
- In 2-ready-for-review/ → Status is "reviewing"
- Etc.

#### Level 2: Metadata JSON Files
Each folder contains a JSON tracking file with details.
- batch-metadata.json (1-generating/)
- review-queue.json (2-ready-for-review/)
- approval-log.json (3-approved/)
- media-log.json (4-with-media/)
- published-log.json (5-published/)

#### Level 3: Master production-queue.md
Central markdown dashboard showing aggregated status across all folders.

### Updating Status (Moving Files)

When a file moves from one folder to another:

1. **Update the metadata file** in current folder (mark as complete)
2. **Move the file** to next folder
3. **Create entry** in next folder's metadata file
4. **Update production-queue.md** with new counts and status

**Automation Note:** This can be scripted. See "Setup Scripts" section below.

### Status Indicators

In production-queue.md, use these indicators:
-  Generating (currently being created)
- ⏳ Ready for Review (awaiting action)
-  In Review (someone is actively reviewing)
-  Approved (passed QC)
-  With Media (media in process)
-  Published (live)
-  Blocked (waiting on external dependency)

## 8. Flow Diagram: File Movement Through Folders

### Standard Linear Flow

```
0-specs/
  (reference)
     ↓
     → 1-generating/ (AI creates content)
           ↓
           → 2-ready-for-review/ (Human reviews)
                 ↓
                 → 3-approved/ (Passes QC)
                       ↓
                       → 4-with-media/ (Media added)
                             ↓
                             → 5-published/ (Live)
```

### With Revision Loop

```
1-generating/
     ↓
2-ready-for-review/ ← Feedback: needs regeneration
     ↓
1-generating/ ← Move back for revision
     ↓
2-ready-for-review/ ← Resubmit
     ↓
3-approved/
     ↓
4-with-media/
     ↓
5-published/
```

### With Major Changes After Approval

```
3-approved/
     ↓
Issue found: content needs major update
     ↓
1-generating/ ← Move back to rework
     ↓
2-ready-for-review/ ← Re-review
     ↓
3-approved/
     ↓
4-with-media/
     ↓
5-published/
```

### Parallel Processing (The Power of This System)

```
Timeline: 10:00 - 14:00

10:00  Batch A starts generating
10:30  Batch A → ready-for-review (human starts review)
       Batch B starts generating (AI working in parallel)
11:00  Batch A → approved (human done, moves to media team)
       Batch B → ready-for-review (AI done, human reviews)
       Batch C starts generating (new AI batch)
11:30  Batch A → with-media (media team working)
       Batch B → approved
       Batch C → ready-for-review
12:00  Batch A → published (live!)
       Batch B → with-media
       Batch C → approved
       Batch D starts generating
```

This shows why the folder system is powerful: work never blocks. Each stage processes files independently while the next stage queues work.

## 9. Common Workflows

### Workflow 1: Standard Flow (Generating → Published)

**Timeline:** ~2 hours per file (typical)

```
Step 1: AI Generation (15-30 min)
 File created in 1-generating/
 batch-metadata.json updated with status
 Production-queue.md shows "Generating"

Step 2: Ready for Review (1 sec)
 File moved to 2-ready-for-review/
 review-queue.json entry created
 Production-queue.md shows "Ready for Review"

Step 3: Human Review (30-60 min)
 Reviewer assigned
 Feedback added (in comments or separate file)
 One of two outcomes:
  A. Approved → move to 3-approved/
  B. Needs changes → move back to 1-generating/ with notes

Step 4: Media Addition (30-45 min)
 File in 4-with-media/
 Media coordinator adds images/videos
 manifest.json created
 Alt text verified

Step 5: Publish (1 min)
 File moved to 5-published/
 published-log.json entry created
 Link to live URL recorded
```

**Automation Note:** Steps 1, 2, 4, and 5 can be scripted. Step 3 (review) requires human judgment.

### Workflow 2: Quick Approve (Pre-approved Spec)

For content types that don't require review:

```
1-generating/
     ↓
2-ready-for-review/ (skip if spec is trusted)
     ↓
3-approved/ (direct move from generating)
     ↓
4-with-media/
     ↓
5-published/
```

**When to use:** Highly structured content (e.g., product descriptions following exact template), AI-generated content from trusted, proven specs.

**Setup:**
- Add `skip_review: true` to batch-metadata.json
- production-queue.md shows direct generation → approved

### Workflow 3: Batch Review (Multiple Files at Once)

For reviewing many files together for consistency:

```
1-generating/
 article-001.md
 article-002.md
 article-003.md 
 article-004.md → All move together
 article-005.md    to 2-ready-for-review/
 batch-metadata.json

2-ready-for-review/
 article-001.md
 article-002.md
 article-003.md 
 article-004.md → Reviewer checks all for
 article-005.md    consistency, quality,
 review-queue.json    tone match

Result: Approve all together
(Or reject all together and regenerate)
```

**Best for:** Collections where pieces relate (series, themed content, etc.)

**Advantage:** Ensures consistency, faster than individual reviews

### Workflow 4: Regeneration (Content Found to Need Rework)

Triggered when issues discovered after approval:

```
Step 1: Move Back to Generating
 File moved from 3-approved/ (or 4-with-media/ or 5-published/)
  back to 1-generating/
 regeneration-notes.md created with feedback

Step 2: Update Request
 batch-metadata.json entry updated with:
  - "status": "regenerating"
  - "reason": "Major factual errors found"
  - "feedback": "Link to regeneration-notes.md"
 Original approval info preserved

Step 3: AI Regenerates
 Reads regeneration-notes.md from previous folder
 Generates new version
 Moves to 2-ready-for-review/ again

Step 4: Re-review (Accelerated)
 Reviewer focuses on changes only
 Approves or requests more work
 Moves to 3-approved/

Step 5: Continue Normal Flow
 → 4-with-media/ → 5-published/
```

**Tracking Regenerations:**

In batch-metadata.json or separate regeneration-log.json:

```json
{
  "article-001.md": {
    "generation_1": {
      "completed": "2025-02-10T09:30:00Z",
      "outcome": "approved",
      "moved_to": "3-approved"
    },
    "regeneration_1": {
      "reason": "Outdated references",
      "requested_by": "reviewer-sarah",
      "requested_date": "2025-02-10T12:00:00Z",
      "completed": "2025-02-10T12:45:00Z",
      "outcome": "approved",
      "moved_to": "3-approved"
    }
  }
}
```

## 10. Tips for Managing Large Batches

### Batch Size Strategies

```
Small Batches (1-3 items)
 Pros: Fast turnaround, easy to manage, quick approvals
 Cons: Less efficiency, more overhead per file
 Use when: Urgent content, complex pieces, first-time specs

Medium Batches (5-10 items)
 Pros: Good balance, parallel processing works well
 Cons: Requires coordination
 Use when: Standard workflow, proven specs, regular cadence

Large Batches (15+ items)
 Pros: High efficiency, good resource utilization
 Cons: Complex coordination, bottlenecks are painful
 Use when: Bulk operations, event content, campaigns

Recommendation: Start with 5-item batches, adjust based on throughput.
```

### Parallel Processing Optimization

**Maximize parallelism by staggering starts:**

```
Timeline (starting times)

10:00  Batch A generation starts (AI)
10:15  Batch A → review queue
       Batch B generation starts (AI)
10:30  Batch A → approval (human reviews)
       Batch B → review queue
       Batch C generation starts (AI)
10:45  Batch A → media queue (media person)
       Batch B → approval
       Batch C → review queue
       Batch D generation starts
11:00  Batch A → publish
       Batch B → media queue
       Batch C → approval
       Batch D → review queue
       Batch E generation starts

Result: At any given time, 5 batches in different stages
Time between publications: ~15 min intervals
Much more efficient than sequential processing
```

### Bottleneck Identification

Monitor production-queue.md for bottlenecks:

```markdown
## Bottleneck Watch

| Stage | Queue Size | Ideal | Status |
|-------|-----------|-------|--------|
```

## 11. Integration with Orchestrator

How an orchestrator (automation script/bot) reads and updates this system:

### Reading Status (Orchestrator → System)

```python
import json
import os
from datetime import datetime

class ProductionQueue:
    def __init__(self, base_path):
        self.base_path = base_path

    def get_status_summary(self):
        """Read production-queue.md and extract status"""
        counts = {
            'generating': len(os.listdir(f"{self.base_path}/1-generating")),
            'review': len(os.listdir(f"{self.base_path}/2-ready-for-review")),
            'approved': len(os.listdir(f"{self.base_path}/3-approved")),
            'media': len(os.listdir(f"{self.base_path}/4-with-media")),
            'published': len(os.listdir(f"{self.base_path}/5-published"))
        }
        return counts

    def get_files_ready_for_review(self):
        """Get all files that moved to review stage"""
        review_queue = f"{self.base_path}/2-ready-for-review/review-queue.json"
        if os.path.exists(review_queue):
            with open(review_queue) as f:
                return json.load(f)
        return {'items': []}

    def get_files_awaiting_media(self):
        """Get all files approved and waiting for media"""
        approved_log = f"{self.base_path}/3-approved/approval-log.json"
        if os.path.exists(approved_log):
            with open(approved_log) as f:
                data = json.load(f)
                return [item for item in data.get('items', [])
                        if item.get('status') == 'approved']
        return []

# Usage
queue = ProductionQueue("6_Session_Working_Files/my-project")
status = queue.get_status_summary()
print(f"Currently generating: {status['generating']}")
print(f"Awaiting review: {status['review']}")
```

### Updating Status (Orchestrator → System)

```python
def move_file_to_next_stage(filename, from_stage, to_stage):
    """Move file and update metadata"""
    import shutil

    from_path = f"{self.base_path}/{from_stage}/{filename}"
    to_path = f"{self.base_path}/{to_stage}/{filename}"

    # Move file
    shutil.move(from_path, to_path)

    # Update metadata in source folder
    self.update_metadata(from_stage, filename, 'completed')

    # Update metadata in destination folder
    self.update_metadata(to_stage, filename, 'received')

    # Update master production-queue.md
    self.update_master_queue()

def update_metadata(self, stage_folder, filename, new_status):
    """Update stage-specific metadata file"""
    metadata_files = {
        '1-generating': 'batch-metadata.json',
        '2-ready-for-review': 'review-queue.json',
        '3-approved': 'approval-log.json',
        '4-with-media': 'media-log.json',
        '5-published': 'published-log.json'
    }

    metadata_path = f"{self.base_path}/{stage_folder}/{metadata_files[stage_folder]}"

    with open(metadata_path) as f:
        data = json.load(f)

    # Find and update the item
    for item in data.get('items', []):
        if item.get('filename') == filename:
            item['status'] = new_status
            item['last_updated'] = datetime.now().isoformat()
            break

    with open(metadata_path, 'w') as f:
        json.dump(data, f, indent=2)

def update_master_queue(self):
    """Update production-queue.md with current counts"""
    summary = self.get_status_summary()

    # Read template
    with open(f"{self.base_path}/production-queue.md") as f:
        content = f.read()

    # Update summary table (regex replacement)
    import re

    new_summary = f"""|  Generating | {summary['generating']} | In Progress |
| ⏳ Ready for Review | {summary['review']} | Awaiting QC |
|  Approved | {summary['approved']} | Awaiting Media |
|  With Media | {summary['media']} | Ready to Publish |
|  Published | {summary['published']} | Complete |"""

    # Replace summary table
    pattern = r'\|  Generating \|.*?\|  Published.*?\|'
    content = re.sub(pattern, new_summary, content, flags=re.DOTALL)

    # Update timestamp
    content = re.sub(
        r'\*\*Last Updated:\*\*.*',
        f"**Last Updated:** {datetime.now().strftime('%Y-%m-%d %H:%M UTC')}",
        content
    )

    with open(f"{self.base_path}/production-queue.md", 'w') as f:
        f.write(content)
```

## 12. Troubleshooting Common Issues

### Issue 1: Files Stuck in 2-ready-for-review/

**Symptom:** Files have been waiting for review for >2 hours

**Diagnosis:**
```bash
# Find oldest files in review
find 2-ready-for-review -name "*.md" -type f -exec ls -lt {} + | head -5
```

**Solutions:**
1. Check review-queue.json for assignment status
2. If unassigned: Assign to available reviewer
3. If assigned: Check reviewer's workload
4. If reviewer overloaded: Assign to another reviewer or bring in temporary help

**Prevention:**
- Set max 3-5 files per reviewer at a time
- Use deadline field in review-queue.json
- Monitor with production-queue.md daily
- Alert if any file in review >90 min

### Issue 2: Media Processing Bottleneck

**Symptom:** Files pile up in 4-with-media/, slow to publish

**Diagnosis:**
```bash
# Check how many files waiting for media
ls 3-approved/*.md | wc -l
ls 4-with-media/*.md | wc -l

# Check age of files in 4-with-media
find 4-with-media -name "*.md" -type f | xargs ls -lt
```

**Root Causes:**
- Insufficient media resources (too few people)
- Unclear media requirements in specs
- Technical issues (compression, optimization)
- Missing assets (need to source images)

**Solutions:**
1. **Immediate:** Prioritize files nearing deadline
2. **Short-term:** Outsource or hire temp media coordinator
3. **Long-term:**
   - Pre-create image templates in 0-specs/
   - Use stock images with clear integration points
   - Document media requirements clearly
   - Consider AI image generation for some content

### Issue 3: Files Lost Between Folders

**Symptom:** Can't find file, unsure which stage it's in

**Prevention (Best):**
1. Always update metadata before moving file
2. Keep manifest of all files in production-queue.md
3. Never manually move files without coordination

**Recovery:**
```bash
# Search all folders
find . -name "article-001.md" 2>/dev/null

# Check metadata files for last known location
grep -r "article-001" */batch-metadata.json */review-queue.json \
  */approval-log.json */media-log.json 2>/dev/null

# Reconstruct from metadata
# Find last status entry and move file there
```

**If Completely Lost:**
```bash
# Find file by date (created/modified recently)
find . -name "*.md" -mtime -1 -type f

# Or search by content
grep -l "specific phrase from content" */*.md
```

### Issue 4: Inconsistent Metadata

**Symptom:** File exists in folder but not in metadata JSON, or vice versa

**Prevention:**
- Write metadata BEFORE moving file
- Use atomic operations (write new file, then move)
- Version control metadata files

**Fix:**
```bash
# List files missing from metadata
for file in 2-ready-for-review/*.md; do
  basename=$(basename "$file")
  if ! grep -q "$basename" 2-ready-for-review/review-queue.json; then
    echo "Missing from metadata: $basename"
  fi
done

# Reconstruct metadata entry from file metadata
ls -l file.md  # get created date
stat file.md   # get all metadata
# Manually add entry to review-queue.json with current timestamp
```

### Issue 5: Wrong File in Wrong Stage

**Symptom:** File approved but somehow ended up back in 1-generating/

**Analysis:**
1. Check batch-metadata.json for entry
2. Was there a regeneration request?
3. Who moved it and when?

**Resolution:**
1. Review regeneration-notes.md if present
2. If no notes: Contact person who moved it
3. Move back to 3-approved/ if regeneration was complete
4. Document what happened in audit notes

### Issue 6: Can't Track Who Did What

**Symptom:** File was approved but no record of who approved it

**Prevention:**
- Always include `approved_by` field in metadata
- Include timestamps
- Track in approval-log.json

**Current Issue:** Add retroactively
```json
{
  "filename": "article-001.md",
  "reviewed_by": "unknown",
  "reviewed_date": "unknown",
  "note": "Retroactively added - no approval record found",
  "recovery_status": "approved_but_undocumented"
}
```

**Future:** Enforce metadata in code/scripts (don't allow file movement without metadata entry)

### Issue 7: publication Date Confusion

**Symptom:** Multiple dates in published-log.json, unclear which is official

**Standards:**
- `published_date`: When file moved to 5-published/
- `canonical_url`: Official live URL
- `approval_chain`: Timestamp for each approval stage
- `archived_dates`: If file was updated later

**Template:**
```json
{
  "filename": "article-001.md",
  "generated_date": "2025-02-10T09:30:00Z",
  "approved_date": "2025-02-10T10:45:00Z",
  "media_complete_date": "2025-02-10T13:00:00Z",
  "published_date": "2025-02-10T15:00:00Z",
  "published_url": "https://example.com/blog/article-001",
  "canonical_url": "https://example.com/blog/article-001",
  "update_history": [
    {
      "updated_date": "2025-02-12T08:00:00Z",
      "archived_version": "5-published/archived/article-001-v1.md",
      "reason": "Minor typo fix"
    }
  ]
}
```

## Quick Reference Cheat Sheet

### Folder Purposes (One Sentence Each)

- **0-specs/:** Single source of truth for what content should contain
- **1-generating/:** AI is creating content here right now
- **2-ready-for-review/:** Content awaiting human QC and approval
- **3-approved/:** Passed review, queued for media resources
- **4-with-media/:** Images/videos added, ready to publish
- **5-published/:** Live content, permanent record

### Key Files to Maintain

| File | Purpose | Update Frequency |
|------|---------|-----------------|
| production-queue.md | Master status dashboard | Every action (or auto-sync) |
| batch-metadata.json | AI generation tracking | As items complete |
| review-queue.json | Review assignments & deadlines | As reviewed |
| approval-log.json | Approval records | As approved |
| media-log.json | Media addition records | As media added |
| published-log.json | Publication history | As published |

### File Movement Checklist

When moving a file from one stage to next:

- [ ] File exists in source folder
- [ ] Update source metadata (mark complete)
- [ ] Move file to destination folder
- [ ] Create entry in destination metadata
- [ ] Update production-queue.md counts
- [ ] Notify stakeholder (if automated, via Slack/email)
- [ ] Verify move was successful

## Summary

This production queue structure enables:

1.  **Clear visibility** - Everyone knows where content is and why
2.  **Parallel processing** - AI and humans never block each other
3.  **Scalability** - Handle 1 file or 1,000 with same system
4.  **Auditability** - Complete record of who did what when
5.  **Automation** - Easy to script and integrate with orchestrators
6.  **Flexibility** - Adapt workflows without breaking structure
7.  **Recovery** - Never lose track of where something is

Start with small batches (5 files), get comfortable with the flow, then scale up. The system will serve you well whether you're managing dozens or thousands of pieces of content.
