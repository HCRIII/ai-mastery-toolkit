# System Wisdom — Quick Start Guide

**Version:** 1.0
**Last Updated:** 2026-02-21

---

## What This System Is

System Wisdom is a behavioral development tracking system built on a simple thesis: the most important variable in human-AI collaboration is not any single output, but the quality of the co-evolutionary relationship itself — and that quality is observable, measurable, and improvable.

The CDPID framework (Collaborative Driven Progressive Intent Discovery) establishes three claims that ground this system. First, intent is emergent — neither party arrives at an interaction with full understanding of what is needed. Second, behavioral development in the dyad is the primary unit of value, not the outputs of any given session. Third, this development follows an observable trajectory that can be taught and accelerated.

System Wisdom is where that trajectory gets tracked.

## What vs. How

The broader AI Mastery Toolkit separates two concerns. Context files and the Context Files track **what** work is being done — the quantity, the projects, the deliverables, the operational state. System Wisdom tracks **how** the work is getting done — the quality of collaboration, the behavioral patterns, the developmental trajectory.

This separation matters because the "how" compounds. A team that improves how they collaborate produces better "what" indefinitely. A team that only tracks what they produce never improves how they produce it.

## The Three Taxonomies

System Wisdom tracks three concurrent behavioral profiles:

**1. Human Behavioral Taxonomy** — Nine dimensions measuring how the human engages with AI: Prompt Aptness, Context Provision, Friction Response, Orchestration Skill, Long-Term Value Awareness, Communication Efficiency, Quality & Completeness Standards, Scalability & Systems Design, and Meta-Cognitive Awareness. Each dimension has five developmental levels (Novice through Expert) with observable behavioral indicators.

**2. AI Behavioral Taxonomy** — Five tiers measuring how the AI performs: Interpretive Behaviors, Generative Behaviors, Adaptive Behaviors, Failure Modes (scored inversely), and Meta-Cognitive Awareness. Each tier has five developmental levels. The AI profile must be bootstrapped each session from files because the AI does not persist between sessions.

**3. Dyadic Dynamic Taxonomy** — Six dimensions measuring the emergent properties of the relationship itself: Orchestral Bandwidth, Collaborative Load Equilibration, Flow Access, Friction Resolution Patterns, Communication Efficiency, and Emergent Capability. These properties belong to neither party — they are properties of the system. Assessed retrospectively, not in real-time.

## How the System Works

### The Core Cycle

```
Work happens → Trigger fires → Snapshot captured → Pattern extracted → Current State updated
```

1. **Work happens.** Normal sessions produce normal outputs. The work is the primary activity — tracking is secondary.

2. **A trigger fires.** Something developmentally significant occurs — a behavioral shift, an emergent insight, a failure mode episode, a flow state transition, a productive friction resolution, a context architecture change, or a research-relevant observation. Either the human or the AI can notice the trigger.

3. **A snapshot is captured.** Using the snapshot template, the observer scores current dimensional levels, notes failure modes, records behavioral observations, and marks the trend direction. This takes 2-5 minutes. The snapshot is timestamped and filed.

4. **Patterns are extracted.** Over time, accumulated snapshots reveal patterns — which dimensions are developing, which failure modes are decreasing, what conditions produce flow states. These patterns move to the System Intelligence folder.

5. **Current State is updated.** The living profile documents are updated to reflect meaningful shifts. These files are what gets read at session startup — they are the AI's bootstrapping context and the human's self-assessment baseline.

### The Five-Step Trigger Protocol

When a trigger condition is noticed:

1. **Notice** — Recognize the trigger condition (behavioral shift, insight, failure mode, etc.)
2. **Flag** — Signal to the other party: "I'm observing something that may warrant a snapshot"
3. **Confirm** — Brief agreement on whether to capture now, later, or not at all
4. **Capture** — Fill in the snapshot template (2-5 minutes)
5. **Resume** — Return to the work thread without losing momentum

The key principle: capturing wisdom should not destroy the flow that produced it.

### What Gets Captured Where

| What | Where | When |
|------|-------|------|
| Timestamped behavioral data points | Snapshot files in taxonomy folders | When triggers fire |
| Current behavioral profiles | Current State files in taxonomy folders | When meaningful shifts accumulate |
| Developmental level definitions | Comprehensive Taxonomy files | When research discovers new territory |
| Validated patterns and principles | 4_System_Intelligence/ | When patterns emerge from multiple snapshots |
| Illustrative episodes | 5_Case_Evidence/ | When episodes demonstrate framework claims |
| Trigger definitions and protocol | 6_Triggers/ | When new trigger conditions are identified |
| Teaching materials and synthesis | 7_Applied_Knowledge/ | When intelligence is ready to be taught |

## Starting a Session

1. Read the three Current State files (Human, AI, Dyadic) to establish behavioral baseline
2. Note any growth edges or active patterns flagged in the profiles
3. Work normally — the tracking system is background awareness, not foreground activity
4. When a trigger fires, follow the five-step protocol
5. At session end, assess whether Current State files need updating

## Key Principles

**The system itself is the evidence.** The tracking system is not separate from the CDPID research — a snapshot at any moment is itself evidence of the framework's claims about observable developmental trajectories.

**The map does not dictate the territory.** Dimensions and tiers are added when research discovers them, not predetermined. The taxonomies grow as understanding deepens.

**Meaning compression over verbosity.** Snapshots should be concise with consistent structure. A well-structured 10-line snapshot is more valuable than a 50-line narrative.

**Portability is a design constraint.** The system should be self-explanatory — hand it to someone unfamiliar with CDPID, and the folder structure, READMEs, and glossary should orient them.

**Frequency follows value, not schedule.** Capture when something significant happens, not on a calendar. High-developmental-velocity sessions might produce multiple snapshots. Routine sessions might produce none.
